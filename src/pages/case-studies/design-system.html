Problems

Working without an established design system introduced problems such as:
  • Inconsistent component behavior and visual patterns.
  • Accessibility considerations being applied inconsistently.
  • Limited documentation of assets, components, and tokens.
  • Poor communication with developers.
  • Slow design process.
  • Designers manually maintained multiple component variants across file.
  • Difficulty onboarding new designers and developers.

Users

This project was not meant to only improve the usability of our components for our users but also to improve the work experience of the design team and development team. Given my background in front-end development and understanding of the challenges facing the team, one of my major goals was to bridge the gap between designers and developers.

Design Strategy

With my directive, we followed the following guidelines:
  • Atomic design by Brad Frost: Adopted this methodology to create reusable, scalable components with clearly defined relationships between UI elements.
  • WCAG 2.2: We strictly followed the guidelines set for accessibility at this level.
  • 4px grid model: Used to this model ensure consistent spacing, alignment, and visual rhythm across all components.
  • Naming convention: Tokens such as icons, colors, and texts follow a fixed naming convention, making them easily identifiable.

Design implementation

Component anatomy

I treated components as modular building blocks. Using the modal component as an example, I structured components according to Atomic Design principles.

States and scenarios

All relevant states and scenarios of each component were presented in design system. This includes but is not limited to: interactive states, valid state, loading state, disabled state and error state.

Accessibility considerations

Accessibility was integrated into each component by default, rather than treated as an afterthought. Some key points were:

  • WCAG-compliant color contrast ratios.
  • Minimum interactive target size of 40×40px.
  • Clear visual hierarchy to support scanability and focus.
  • Clear patterns for visual association i.e for example having consistent error and validation feedback design.

Development implementation

Components were built to be WCAG compliant and respect semantic HTML design. I personally built some complex components with the development team using Radix UI component library. The library provides components that follow WAI-ARIA accessibility standards, keyboard navigation, and screen reader compatibility out of the box.

Outcome

We either completely resolved or significantly reduced the problems listed above. Apart from the benefit of an informed and consistent UI design, one important takeaway from this project was how good documentation thats includes the reasoning behind important decisions helped to solidify the mission of the company and make each member of the team more confidently autonomous. Errors or gaps in design was now much easier to spot and correct by even the developers.
